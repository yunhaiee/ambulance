# 연구실 발표 Q&A 대비 (2026-07)
숫자 카드는 맨 아래. 원칙: 모르면 지어내지 말고 "좋은 지적입니다, 파일럿 설계에 반영하겠습니다"로 받기.

## A. 신뢰성·안전 (가장 세게 들어옴)

**Q1. LLM이 KTAS를 오판하면? 사람 죽으면 누구 책임인가?**
이 시스템에서 LLM의 KTAS는 "보조 판단"이고 최종 이송 결정과 처치는 구급대원에게 있습니다(human-in-the-loop). 더 중요한 건 판단의 분업 구조입니다 — 병원 수용 여부는 LLM이 아니라 코드에 박힌 규칙(수술실≥1 ∧ 병상≥1 ∧ 수술가능 의사≥1)이 실시간 DB 조회로 결정합니다. 모든 결정은 감사로그(JSONL)에 단계별로 남아 사후 추적이 가능합니다. KTAS 정확도 자체는 골든셋 기반 회귀평가(adk eval)를 다음 단계로 설계해뒀습니다.

**Q2. 환각은 어떻게 막나?**
아키텍처로 막습니다. 숫자·거리 정렬·수용 판단·메시지 조립은 전부 결정론적 코드이고, LLM은 라우팅과 언어 인터페이스만 맡습니다. 실제로 1년 전 버전은 220줄 프롬프트로 "환각 금지"를 빌었는데, 업그레이드에서 그 로직을 코드로 옮기자 프롬프트가 31줄이 됐고 해당 오류 클래스가 사라졌습니다. 도구 실패는 전부 isError로 표면화되어 LLM이 조용히 지어내는 경로 자체가 없습니다.

**Q3. LLM이 도구 호출을 빼먹거나 잘못 호출하면?**
실제로 겪었습니다. flash가 긴 여러 줄 텍스트 인자를 쓰다 MALFORMED_FUNCTION_CALL을 내는 걸 라이브에서 발견했고, 해법은 "짧은 필드 4개짜리 전용 도구 + 포맷팅은 코드"로 재설계였습니다. 이후 재현 0회. 도구 인터페이스 설계 원칙(짧은 인자, 조립은 코드)을 얻은 게 성과입니다.

**Q4. 병원이 수용 응답했는데 도착 시점엔 병상이 없다면?**
현재 판단은 문의 시점의 실시간 DB 스냅샷입니다. 도착 시점 보장은 예약(hold) 프로토콜이 필요하고, 이건 파일럿에서 병원과 함께 설계할 항목입니다. 다만 지금도 이중 안전장치로 "구급대원 직접 확인 전화" 안내를 함께 냅니다.

**Q5. 시스템이 죽으면?**
우아한 강등(graceful degradation)이 설계 원칙입니다. 에이전트 미연결 병원은 no_agent로 표시되어 "직접 전화" 안내가 나가고, 타임아웃·오류도 전부 구조화된 상태로 구급대원에게 전달됩니다. 최악의 경우에도 현행 방식(전화)으로 자연 복귀하며, 그 사이 KTAS 판단과 병원 목록은 이미 확보된 상태입니다.

## B. 평가·검증 방법론

**Q6. "50초"는 몇 번 측정한 수치인가? 분산은?**
정직하게: 완주 실행 수 회의 실측(43~54초 범위)이고, 무료 API 티어 + gemini-2.5-flash 조건입니다. 통계적 주장이 아니라 "실현 가능성의 증거"로 제시합니다. 다만 모든 실행이 단계별 타임스탬프로 감사로그에 남는 인프라가 있어, 대규모 벤치마크는 준비돼 있습니다.

**Q7. KTAS 판단 정확도는 평가했나?**
아직입니다. 현재는 프롬프트 기반이고, 골든셋(증상→KTAS 등급·진료과 정답 세트) + adk eval 회귀 테스트가 로드맵 최우선 항목입니다. 의료진 검증 협력이 필요한 부분이라 연구실에서 도움을 받고 싶은 지점이기도 합니다.

**Q8. 베이스라인(사람이 전화)과 비교 실험은?**
직접 비교는 파일럿 과제입니다. 현재 근거는 간접 지표 — 뺑뺑이 사례들의 수용병원 확정 소요(수십 분~시간대)와 저희 실측 50초의 차이, 그리고 구급대원 설문(94%가 현행 상황판 개선 필요)입니다.

**Q9. 병원 데이터가 Notion 모형인데 실제와의 갭은?**
인정합니다. 다만 수용판단 모듈(capacity.py)이 어댑터 구조라 데이터 소스만 교체하면 됩니다. 1차로 국립중앙의료원 실시간 가용병상 공공 API(공공데이터포털) 연동, 2차로 파일럿 병원의 실데이터 어댑터가 계획입니다.

## C. 아키텍처·기술 선택

**Q10. 왜 멀티에이전트인가? LLM 하나에 도구 다 붙이면 되지 않나?**
조직 경계가 곧 에이전트 경계이기 때문입니다. 병원·보험사는 자기 데이터를 외부에 내주지 않고 자기 인프라에서 자기 에이전트를 운영하며(데이터 주권), 표준 프로토콜(A2A)로만 대화합니다. 단일 LLM 구조는 모든 기관 데이터를 한곳에 모아야 해서 이 도메인에선 성립하지 않습니다. 부수 효과로 장애 격리와 독립 배포도 얻습니다.

**Q11. A2A가 왜 필요한가? 그냥 REST API면 되지 않나?**
REST로 하면 기관마다 스펙 협상을 해야 합니다. A2A는 에이전트 카드로 능력을 표준 발견하고, 태스크 수명주기·스트리밍·인증 스킴이 규격화돼 있습니다. 작년에 Linux Foundation으로 이관되며 업계 표준이 됐고 — 1년 전 저희 베팅이 맞았습니다. 저희는 실제로 0.2→0.3 마이그레이션과 SSE 스트리밍 전환까지 겪으며 실무 검증을 했습니다.

**Q12. 전부 MCP로 하지 왜 일부는 일반 함수인가?**
저희가 얻은 설계 기준입니다: 외부에 표준 인터페이스로 노출할 도구(카카오 교통)는 MCP, 에이전트 내부 로직(Slack 전송, Notion 조회)은 일반 함수가 낫습니다. 실제로 아카이브된 MCP 서버 의존을 걷어내고 직접 호출로 바꾸면서 안정성이 올라갔습니다. "MCP는 목적이 아니라 경계에 쓰는 도구"라는 게 배운 점입니다.

**Q13. 왜 Gemini인가? Claude나 GPT는?**
프레임워크(Google ADK)의 네이티브 경로 + 속도·비용(flash 계열) 때문입니다. 다만 모델은 환경변수 하나로 교체 가능하게 설계했고, LiteLLM 래퍼로 Claude 등도 붙습니다. 실서비스라면 KTAS 판단 노드만 상위 모델로 올리는 하이브리드가 저희 권장안입니다.

**Q14. 병원 문의를 병렬로 바꿀 때 어려움은?**
세 가지였습니다: 병원별 타임아웃 예산(45초), 카카오 지명↔에이전트명 퍼지 매칭, 그리고 응답을 accept/decline/no_agent/timeout으로 구조화해 LLM이 아닌 코드가 집계하게 한 것. 결과적으로 전체 소요가 "가장 느린 병원 하나" 수준으로 떨어졌습니다.

**Q15. 에이전트 간 메시지가 자연어면 스키마 없는 것 아닌가?**
혼합입니다. 사람이 읽는 부분은 자연어, 기계가 파싱하는 부분은 JSON 계약(예: 병원 응답의 decision 필드)입니다. LLM이 형식을 흔드는 경우를 대비해 파서를 다층화(펜스 JSON → 무펜스 JSON → 한/영 마커)했습니다. 완전 스키마 강제는 A2A 확장으로 가능하지만, 자연어 병행이 이종 기관 온보딩을 쉽게 만든다는 트레이드오프를 택했습니다.

## D. 보안·프라이버시

**Q16. 환자정보가 외부 LLM API로 나가는데 개인정보보호는?**
현재는 가상 환자 데모입니다. 실서비스 설계는: 국내 리전 Vertex AI(데이터 미학습 계약), 개인정보보호법 15조1항5호(응급 시 수집 근거), 최소수집 원칙, 그리고 이미 구현된 로그 PII 마스킹 + 감사추적입니다. 법적 근거는 1년 전 검토를 유지하고, 기술적 안전장치를 올해 실장했습니다.

**Q17. Slack으로 환자정보를 보내는 게 말이 되나?**
데모용 프록시입니다. "기관 알림 채널"의 자리표시자로, 실서비스에선 병원 EMR 인터페이스와 보험사 시스템 직결로 대체됩니다. 아키텍처상 전송 도구만 교체하면 되는 구조입니다.

**Q18. 에이전트 사칭이나 프롬프트 인젝션 공격은?**
전 구간 bearer 토큰 상호인증을 구현했고(무인증 요청 401), 카드 서명·mTLS는 A2A 표준 로드맵과 함께 갑니다. 프롬프트 인젝션은 공격 표면 자체를 줄였습니다 — 각 에이전트 도구가 최소권한이고, 수용 판단이 코드라서 "주입된 문장이 판단을 바꿀" 경로가 좁습니다. 완전하진 않으며, 입력 검증 레이어가 향후 과제입니다.

## E. 확장성·현실 적용

**Q19. 병원 500개, 5000개로 확장되나?**
병원 에이전트는 코드 1벌 + 설정파일 N개 구조라 추가 비용이 설정 1개입니다. 런타임은 상위 N개 병원만 병렬 문의하고, 전국 스케일에선 공공 가용병상 API가 1차 필터, 에이전트 문의가 확정 레이어가 됩니다. 서버는 무상태에 가까워 Cloud Run 수평 확장이 계획입니다.

**Q20. 병원이 왜 참여하나?**
응급 전화 응대는 병원에도 인건비입니다. 에이전트가 1차 응대를 대신하고 병원 내부엔 슬랙(→EMR) 자동 보고가 남습니다. 정책적으로는 응급의료기관 평가와 연계하면 참여 유인이 생기고, 참여 장벽은 "에이전트 하나 세우기"로 최소화했습니다.

**Q21. 기존 중앙응급의료센터 상황판(NEDIS)과의 관계는?**
대체가 아니라 결합입니다. 공공 데이터를 후보 필터로 쓰고, 에이전트는 그 위에서 "지금 이 환자를 받을 수 있는가"라는 확정 질문을 답합니다. 94%가 개선을 원한 건 상황판 데이터와 현장의 갭 — 저희가 메우는 게 바로 그 갭입니다.

**Q22. 음성 인식은? 시끄러운 구급차에서 STT가 되나?**
Gemini Live API 연동이 다음 단계입니다. 구급차 소음 환경은 정직하게 난제이고, 노이즈 강건성 검증은 파일럿의 핵심 실험 항목입니다. 현재 텍스트 파이프라인은 음성이 붙어도 그대로 재사용됩니다(입력 모달리티만 교체).

## F. 연구 관점

**Q23. 엔지니어링이지 연구적 novelty가 있나?**
세 가지로 답합니다. (1) 조직 간(cross-organizational) 멀티에이전트 프로토콜의 공공 응급 도메인 실증 — A2A 실전 사례 자체가 아직 희소합니다. (2) LLM-결정론 하이브리드 판단 구조: 어떤 판단을 LLM에, 어떤 것을 코드에 둘 것인가의 설계 원칙과 그 실패 사례들. (3) 전 과정 감사로그 기반의 에이전트 시스템 평가 방법론. 셋 다 논문화 가능한 질문이라 생각하고, 연구실의 시각을 듣고 싶습니다.

**Q24. 만들면서 가장 크게 배운 것/실패는?**
"통합 라이브 테스트 없이는 못 찾는 버그"입니다. 3일간 실 API로 돌리며 6개를 찾았습니다 — Notion 스키마 불일치, A2A 비동기 응답을 50ms에 읽어버린 문제(폴링→SSE 전환), 함수호출 형식 깨짐, 영어 응답 파싱, 프레임워크의 한글 앱명 거부, 이벤트 루프 오염. 전부 유닛 테스트로는 안 잡히는 통합 계층 버그였습니다.

**Q25. 1년 전과 무엇이 다른가 한 줄로?**
"주장이 실측이 됐고, 프롬프트가 코드가 됐고, 데모가 시스템이 됐습니다." — 1분 목표→50초 실측, 220줄 프롬프트→31줄+결정론 코드, 복사-붙여넣기 병원 2개→설정파일 N개 구조.
